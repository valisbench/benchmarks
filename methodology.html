Methodology

What this benchmark is

A pre-registered evaluation of frontier language models on a specific failure mode. Before any model is called, we publish the dataset (or its hash), the model roster with exact version strings, the sampling parameters, the scoring rubric, and the hypothesis. The signature is anchored to two transparency logs that we do not operate. After the run, we publish the results on the same page as the manifest. If results are null or unfavorable, they publish anyway.

Why the form matters

Most public benchmarks publish numbers, not commitments. A benchmark whose dataset can be silently changed, whose roster can be re-rolled, or whose unflattering runs can be hidden is not a measurement instrument. It is a press release with a confidence interval.

The form we use here is intended to make those moves visible. The manifest hash is in Rekor. The Rekor entry records the identity of the GitHub Actions workflow that signed it. The OpenTimestamps proof anchors the manifest hash to a Bitcoin block. If we re-publish a different manifest under the same version tag, the old one stays in Rekor forever.

What v1 measures

v1 ships two wedges, both synthesizable without external labelers:

  • Phantom precision. The model is shown a paragraph of plausible business prose that contains one fabricated quantitative claim plus one to three real, sourced numerical claims as distractors. The model is asked to flag the fabricated claim. Scored as F1 against ground-truth fabricated spans, with the distractor false-positive rate as a separately-reported secondary metric.
  • Mechanism-gap. The model is shown a single "X because Y" sentence. The model must classify whether Y is a supported mechanism for X. Scored as binary F1, with recall on unsupported as the headline number.

Refusal handling

A refusal counts as a separate category, not as a false negative or false positive. Refusal rates are reported per model per benchmark. F1 is computed over the non-refused subset. The full sample-size denominator is shown alongside. A model that refuses every case scores "refusal: 100%" with no F1 reported, which is the correct answer.

Outcome categories

Every model call resolves to exactly one of seven categories:

  • response -- model returned a parseable answer
  • refusal -- model declined to answer
  • partial_response -- response cut off at max_tokens or a stop sequence
  • error_timeout -- per-case timeout exceeded
  • error_rate_limit -- 429 from provider after retries exhausted
  • error_other -- unclassified error
  • malformed -- response present but did not match the expected schema

How verification works

Each published version has three companion files in the public repo:

  • manifest.yaml -- the pre-registered specification, the signed bytes
  • manifest.bundle.json -- the Sigstore signature plus the Rekor entry
  • manifest.yaml.ots -- the OpenTimestamps proof

A third party verifies these by running cosign and ots against them. The verify.sh script in the public repo has zero VALIS code paths. A Debian Docker container with cosign and ots installed runs it. See the verify page.

External methodology review

Each publication is reviewed by one externally-named methodology reviewer. The reviewer is paid for an honest critique, not a friendly write-up. Their written assessment publishes verbatim alongside the results, with their name on it. A refusal to sign is itself published.

What v1 does not yet do

  • Datasets are synthesized, not human-labeled. Phantom-precision tier-3 cases pass founder spot-check (N=50) before publication. v2 will add external labelers for a gold subset.
  • The private held-out partition (200 cases per wedge) is encrypted at rest in the public repo. The decryption key is held by the founder. v2 will define a rotation policy and a key escrow mechanism.
  • Determinism is best-effort across providers. temperature=0 and a fixed seed do not guarantee identical bytes across re-runs on every provider; the manifest documents this honestly.
