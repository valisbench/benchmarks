Methodology

How VALIS Benchmarks work.

Most public AI benchmarks publish numbers, not commitments. A benchmark whose dataset can be silently changed, whose roster can be re-rolled, or whose unflattering runs can be hidden is not a measurement instrument. It is a press release with a confidence interval.

We do it differently. Before any model is called, we publish the dataset hash, the model roster with exact version strings, the sampling parameters, the scoring rubric, and the hypothesis. The manifest is signed with Sigstore cosign. The signature is anchored to the Sigstore Rekor transparency log. The hash is timestamped to a Bitcoin block via OpenTimestamps. If we re-publish a different manifest under the same version tag later, the old one stays in Rekor forever.

After the run, the results publish on the same page as the manifest. Null results publish. Unfavorable results publish. The commitment to do that is itself anchored.

The seven principles

One. Pre-registration is the artifact. The manifest freezes everything that could be retroactively changed: dataset hash, model versions, temperature, seed, scoring rubric, refusal policy, failure-publication commitment. The signed bytes of that file are the benchmark.

Two. Two anchors, not one. The Sigstore Rekor log records the GitHub Actions identity that signed the manifest. The OpenTimestamps proof anchors the manifest hash to a Bitcoin block. Re-publishing under the same tag is detectable in either log alone.

Three. Refusal is a separate category, not a wrong answer. We report refusal rates per model per wedge. F1 is computed over the non-refused subset. A model that refuses every case scores "refusal: 100%" with no F1 reported, which is the correct answer.

Four. Synthetic ground truth is disclosed, not hidden. Where the dataset is synthesized (no human labels), we say so on the wedge page. Where the rubric uses a constructed dependency graph instead of expert judgment, we say so on the wedge page. The disclosure is part of the cryptographic commitment.

Five. Spot-check panels are required, not optional. Each wedge's case set is reviewed by three independent frontier models before publication: Claude Opus 4.7, GPT-5.5, and Gemini 3.1 Pro. The gate is zero unanimous fails on a 50-case sample. The panel result is published verbatim alongside the manifest.

Six. Disagreement is data. Where the three reviewers diverge on rubric application, we publish the cases verbatim with each reviewer's rationale. The boundary of a rubric is itself part of the published artifact.

Seven. Anyone can verify. The verify.sh script in the public repo has zero VALIS code paths. A Debian Docker container with cosign and ots installed runs it. You do not need to trust VALIS to verify a VALIS benchmark.

Worked example: phantom_precision end-to-end

To make this concrete, here is how a single wedge moves from pre-registration to anchored result.

Step one. The wedge dataset is constructed. For phantom_precision, we author a paragraph of plausible business prose containing one fabricated numerical claim ("a recent McKinsey study showed 42% gains") and one to three real numerical claims from sourced documents as distractors. Each case is reviewed by the founder for plausibility. The cases file is hashed.

Step two. The manifest is written. It fixes the dataset SHA256, the model roster (Claude Opus 4.7, GPT-5.5 snapshot gpt-5.5-2026-04-23, Gemini 3.1 Pro Preview), sampling parameters (temperature 0, seed 42, top_p 1, max_tokens 1024), the scoring rubric (per-passage F1 against fabricated spans), the refusal policy (refusals as a separate category), and the failure-publication commitment.

Step three. The spot-check panel runs. Three frontier models review a 50-case sample. Each model returns pass/fail per case with rationale. We collate the verdicts. Phantom_precision v1.0.0 passed the gate with zero unanimous fails. Forty-four cases were unanimous pass; four were 2-of-3 pass; two were 1-of-3 pass. The four and two together are the disagreement set, published verbatim.

Step four. The git tag is pushed. benchmarks-v1.0.0-phantom_precision. The GitHub Actions workflow picks up the tag, runs cosign sign-blob with keyless OIDC against the manifest, posts the signature to Rekor, receives back the log index, and runs ots stamp against the manifest hash. The OpenTimestamps proof is committed to the public repo in "pending" state.

Step five. Within 3-6 hours, the Bitcoin block lands. The daily ots_finalizer workflow upgrades the proof to "confirmed" automatically. The manifest hash is now anchored in a Bitcoin block. The record cannot be edited.

Step six. The verifier runs. Anyone with cosign, ots, and the public repo can run ./verify.sh v1.0.0. The script computes the manifest's SHA256, looks up the Rekor entry by log index, verifies the signature against the recorded GitHub Actions identity, and verifies the OpenTimestamps proof against the Bitcoin block. A FULL PASS exits 0.

Sampling parameter deviation

Deviation from pre-registered manifest

The v1.0.0 manifests pre-register temperature: 0, top_p: 1, seed: 42 for all three providers. After pre-registration and anchoring, Anthropic deprecated these parameters on Claude Opus 4.7 and OpenAI deprecated them on GPT-5.5; both APIs reject non-default values and use internal adaptive sampling. The harness omits sampling parameters when calling these models. Gemini 3.1 Pro Preview still accepts the parameters and was called with the pre-registered values.

Anthropic's own documentation states that temperature=0 never guaranteed identical outputs on prior models. The substantive commitment of the pre-registration -- no stochastic tuning of model behavior -- is honored across all three providers. The deviation is in which API surface expresses that commitment, not in the commitment itself.

Reasoning-model token budgets

All three frontier models on this version's roster -- Claude Opus 4.7, GPT-5.5, and Gemini 3.1 Pro Preview -- are reasoning models. They do internal thinking before producing the final answer, and that thinking consumes part of the max_tokens budget on each call. The manifests pre-register max_tokens: 1024 for every model.

Gemini 3.1 Pro Preview exposes its thinking content as part of the response stream by default; when the budget is tight, the budget is consumed before the answer arrives and the case is recorded as partial_response rather than response. Excluded from F1, the same as any non-response outcome. Anthropic and OpenAI's reasoning content is omitted from the response stream by default, so the same 1024 budget produces the answer directly.

This is a measurement observation about how each model handles the budget it is given, not a deviation from pre-registration. The outcome_counts field of every result row reports the per-case breakdown explicitly. Where you see a high partial_response rate, that is itself data about the model.

What this version does not yet do

  • Datasets are synthesized, not human-labeled. Spot-check panels confirm internal consistency, but v2 will add external labelers for a gold subset.
  • The private held-out partition (200 cases per wedge) is encrypted at rest in the public repo. The decryption key is held by the founder. v2 will define a rotation policy and a key escrow mechanism.
  • Determinism is best-effort across providers. Temperature zero and a fixed seed do not guarantee identical bytes across re-runs on every provider. The manifest documents this.
