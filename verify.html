How to verify a published version

Verification confirms two things: that the manifest's signature is valid (it was signed by the GitHub Actions workflow that ran when the version was tagged), and that the manifest was anchored to the Sigstore Rekor transparency log and to OpenTimestamps. The verify.sh script that does this has zero VALIS code paths. You do not need to trust VALIS to verify a VALIS benchmark.

Requirements

  • cosign v2.5 or newer (Sigstore CLI)
  • ots (opentimestamps-client) -- pip install opentimestamps-client
  • jq, sha256sum (or shasum -a 256 on macOS), git

Procedure

git clone https://github.com/valisbench/benchmarks.git
cd benchmarks
./verify.sh v1.0.0

Exit codes:

  • 0 -- full pass. Cosign valid, Rekor entry verified, OTS Bitcoin-confirmed.
  • 1 -- partial pass. Cosign valid, Rekor entry verified, OTS still pending. Bitcoin confirmation can take three to six hours after publication; a partial pass within that window is not a failure.
  • 2 -- fail. Cosign or Rekor verification failed. The manifest was altered after publication or the signature is invalid. Open an issue.
  • 3 -- fail. A required input file is missing from the repo. The publication itself was incomplete. Open an issue.

From a clean Docker container

The public repo ships a Dockerfile that builds a debian:bookworm-slim image with only the verifier dependencies. No VALIS code is installed in the image, and the build fails if any file in the verify path imports from valis.*.

docker build -t valisbench-verify .
docker run --rm valisbench-verify ./verify.sh v1.0.0

What "pass" means

A passing verify means: the manifest YAML you are reading is the exact bytes that were signed at publication time, the signature was issued by the GitHub Actions workflow whose identity is recorded in the Rekor log, and the Rekor entry is itself verifiable against the Sigstore transparency log. For full pass, the manifest hash is also anchored to a Bitcoin block via OpenTimestamps.

What it does not mean: that the methodology is sound, that the dataset is well-constructed, that the results are interesting, or that the model performances are stable across re-runs. Verification is about provenance, not quality. For methodology critique, read the external reviewer notes published alongside each version. For result interpretation, read the per-version results page.
